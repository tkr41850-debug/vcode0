# Phase Review — Opus, holistic architecture

This review examines all five phase docs for cross-cutting consistency. Phases were drafted in parallel and the seams show. Most issues stem from one of three patterns: (a) silent contradictions between phase decisions made independently; (b) hard problems pushed forward to phases that never explicitly pick them up; (c) migration/numbering coordination that is acknowledged in prose but not actually settled.

## Critical issues (block landing)

- **Phase 2 vs Phase 3 — session-storage decisions directly contradict each other.** Phase 2 step 2.3 (`phase-2:238-317`) commits *centralized* session storage on the orchestrator: `RemoteSessionStore` proxies `save`/`load`/`saveCheckpoint` over IPC, the orchestrator becomes the authoritative session host, and the rationale explicitly says "phase 5's lease/takeover work needs this to be true". Phase 3 step 3.4 (`phase-3:147,156`) then introduces a *per-run worker-side* `FileSessionStore` rooted at `<workerScratch>/sessions/<agentRunId>/` and explicitly distinguishes "orchestrator-side store stays at `.gvc0/sessions/`; the worker-side store is per-run." Phase 3 step 3.5 (`phase-3:178,184`) goes further — its sticky-resume fallback hinges on "the session lives on the dead worker." These two designs cannot both be true. **Suggested fix:** delete the worker-side `FileSessionStore` parameterization from phase 3 step 3.4 entirely (sessions live on the orchestrator per phase 2); rewrite phase 3 step 3.5 to drop the "session is gone with the worker" reasoning — sticky resume becomes a cache-locality optimization only, not a correctness requirement, and `not_dispatchable: 'unknown_worker'` falls through to *another worker with the same session id*, not to `mode: 'start'`.

- **Phase 3 vs Phase 5 — ownership column duplicates the lease table for the same data.** Phase 3 step 3.1 (`phase-3:39-55`) adds `owner_worker_id` and `owner_assigned_at` columns to `agent_runs` and *explicitly rejects* a `run_assignments` table on the rationale "ownership is 1:1 with a run at any moment in time and the existing row already carries every other lifecycle field." Phase 5 step 5.1 (`phase-5:163-180`) then introduces exactly that rejected `run_leases` table, keyed `agent_run_id PRIMARY KEY` (still 1:1), with `worker_id`, `granted_at`, `expires_at`, `state`. Now the system has *two* sources of truth for "who owns this run": the column on `agent_runs` and the row in `run_leases`. Phase 5 never says what happens to `owner_worker_id`. **Suggested fix:** explicitly state in phase 5 step 5.1 that `agent_runs.owner_worker_id` is dropped (along with `owner_assigned_at`) and replaced by the lease table; or, alternatively, push back to phase 3 and have step 3.1 *not* add the column, instead persisting ownership through an interim table that becomes `run_leases` in phase 5. As written, step 5.9's column-drop migration retires `worker_pid`/`worker_boot_epoch` but leaves `owner_worker_id` orphaned forever.

- **Phase 1 vs Phase 3 — migration numbering collision is "coordinated" but not resolved.** Phase 1 step 1.3 (`phase-1:159`) uses migration **012** for `workers`, citing baseline's reservation of 010 (ipc_quarantine) and 011 (inbox_items). Phase 3 step 3.1 (`phase-3:39,45`) uses migration **010** for `agent_run_owner_worker` with the dodge: "if `01-baseline` has not yet shipped its 010, this phase claims `010` and `01-baseline` re-numbers in its own track." This means after all three tracks land, the actual migration sequence depends on landing order. If 02-distributed phase 1 ships first you get 012, then phase 3 lands 013 (not 010), then baseline phase 1 is stuck — its 010 is taken. Phase 3 step 3.6 (`phase-3:217`) compounds this by claiming **011** for an index. **Suggested fix:** pin a hard numbering plan now: phase 1 = 012, phase 3 = 013 (owner column) and 014 (index), phase 5 = 015 (run_leases) and 016 (drop columns). Update each phase doc's migration filename and DB-registration line. The "renumber later" hand-wave will produce a partial-deploy unreachable state if any phase lands out of expected order.

- **Phase 4 vs Phase 1 — phase 1 explicitly says workers are "not yet dispatchable" but phase 4 needs feature-phase dispatch to a registered remote worker, which phase 2 only landed for tasks.** Phase 1 step 1.5 (`phase-1:222`) says remote workers "do *not* appear in `LocalWorkerPool.idleWorkerCount()` and **do not** receive any `dispatchRun` call." Phase 2 step 2.5 (`phase-2:381-452`) lifts this for *task* dispatch via `MultiHarnessPool` selecting by `worker.capability`. Phase 4 then assumes that the same `dispatchRun` plane carries `feature_phase` scopes to remote workers (`phase-4:30-33`), but never says whether the phase-1 capability declaration (`scopeKinds: ['task', 'feature_phase']` from `phase-1:57`) actually filters dispatch — phase 2's `MultiHarnessPool` only routes by worker kind (local-spawn vs remote-ssh), not by scope kind. A worker that registers with `scopeKinds: ['task']` will silently receive a `feature_phase` dispatch in phase 4. **Suggested fix:** phase 4 step 4.3 must state explicitly that the dispatcher filters worker candidates by `scopeKinds.includes(scope.kind)` before selection, and add a test that a task-only worker rejects feature-phase dispatch. Phase 1 step 1.1 already declares the capability — phase 4 just has to honor it.

- **Phase 4 step 4.5 — proposal-host network seam is an unbounded ordering problem.** Phase 4 step 4.5 (`phase-4:158-184`) lifts `GraphProposalToolHost` onto the worker, streaming each tool-call mutation as a `proposal_op` IPC frame to the orchestrator, with `proposal_submitted` as the checkpoint. The "ordering must be strict" review prompt (`phase-4:182`) acknowledges the risk but the design has no actual ordering guarantee: WebSocket frames are ordered per-connection, but the worker's faux-model agent loop can `await tool.run()` and emit ops faster than the orchestrator's `proposalOpSink` can drain them, and a takeover (phase 5) can interrupt mid-stream. The orchestrator-side adapter has no sequence number; if a frame is dropped during reconnect, the TUI mirror diverges silently. The phase 5 design says progress frames are *not* fence-checked (`phase-5:378`) — but `proposal_op` is essentially state-mutating and must be fence-checked, otherwise a reconnected stale worker can replay ops on top of the new lease. **Suggested fix:** phase 4 step 4.5 should add a per-run monotonic `seq` to `proposal_op` and `proposal_submitted`; phase 5 step 5.5 should classify these frames as state-mutating and require `fence` (currently it only lists `result`/`error`/`claim_lock`/`request_help`/`request_approval`). Phase 5's review subagent prompt at `phase-5:412` should add `proposal_op` / `proposal_submitted` / `proposal_phase_ended` to the fenced-frames list.

- **Phase 5 step 5.5 — `liveRuns` cache invalidation on takeover has a TOCTOU window.** Phase 5 step 5.5 (`phase-5:381-385`) caches `agent_runs.fence_token` per dispatch in `liveRuns`; takeover invalidates the cache "via the scheduler event from step 5.4." But phase 5 step 5.4 (`phase-5:319-345`) runs the sweep on a separate timer (`sweepIntervalMs`); between `expireLease`'s fence bump and the cache invalidation event reaching the IPC handler, the old worker can land a `result` frame with the now-stale-but-cached fence and the orchestrator will accept it. Worse, after takeover, the *new* worker's first `result` frame must arrive with the new fence before the cache is updated — but the cache is keyed off "stored at dispatch time," so the new dispatch refreshes it correctly only if grant happens before any frame from the new worker is processed. **Suggested fix:** drop the `liveRuns` fence cache; `updateAgentRunTxn`'s `expectedFence` check (`phase-5:391-396`) reads from the source of truth, which is fine. The cache is a perf optimization that introduces a race; without it, the source-of-truth check on every mutating frame is bulletproof.

- **Phase 5 step 5.7 (orchestrator crash, scenario 1) — reattach is hand-waved as "phase-1 reconnect handshake."** Phase 5 step 5.7 (`phase-5:482-491`) says that on orchestrator restart with non-expired leases, runs "reattach via phase-1 reconnect... lease row not deleted, fence not bumped." But phase 1 only landed *registration* and *heartbeat* — it never specified a "reconnect" handshake that preserves run ownership across orchestrator restart. Phase 1 step 1.4 (`phase-1:187`) puts every connection in pre-register state until a `register` frame arrives, then runs are dropped. The worker has no way to tell the orchestrator "I'm currently running run X with fence F" on reconnect; phase 5 step 5.3 only adds `leases` to *outgoing* heartbeats from the worker, not to a recovery handshake. So after orchestrator restart, the orchestrator has no in-memory map of `agent_run_id → ws connection`, and no protocol for the worker to volunteer one. The first heartbeat *will* carry the `leases` array, but the orchestrator's IPC handler must already know the connection→workerId binding before it can route the frames — and that binding only forms on `register`. **Suggested fix:** phase 1 step 1.2 should add a `reconnect` frame variant carrying `bootEpoch` + held `agentRunId`s (or phase 5 step 5.3 should make the first post-restart heartbeat carry full reattach info and define reattach semantics explicitly). Without this, the entire "scenario 1" of step 5.7 is untestable.

- **Phase 5 — fence-token persistence ordering is wrong relative to dispatch.** Phase 5 step 5.2 (`phase-5:218-238`) grants the lease *after* the harness returns a `SessionHandle`, with `fence: run.fenceToken`. But the dispatch payload was already sent to the worker (the worker is what made the harness return). The worker now holds an in-flight run *without* knowing its fence — it cannot stamp outgoing frames with the right fence until step 5.3's heartbeat round-trips. **Suggested fix:** the fence must be communicated to the worker on the *initial* `run` frame, not after. That means: (a) read+increment `agent_runs.fence_token` *before* dispatch (the orchestrator owns it); (b) send fence in the `run` frame payload; (c) only then call `harness.start`; (d) call `grantLease` immediately after. Step 5.5's frame-validator already expects the worker to send fence on every state-mutating frame (`phase-5:374-380`) — it has to come from somewhere, and "from the run frame the orchestrator sent on dispatch" is what step 5.3 says (`phase-5:281-282`). Step 5.2's ordering contradicts that.

## Significant issues (worth addressing)

- **Phase 2 step 2.5 — `RuntimePort` claims to stay stable but `MultiHarnessPool` adds capacity-aware routing prematurely.** Phase 2 step 2.5 (`phase-2:401`) says "`RuntimePort` surface is unchanged — phase 2 keeps the seam stable." But the same step also says the dispatcher "asks the registry for the next available worker (single worker for now — multi-worker is phase 3)" and "reads its capability". This is precisely the worker-aware-dispatch concern phase 3 step 3.2 (`phase-3:67-99`) is supposed to introduce. Phase 2's `MultiHarnessPool` either is or isn't capacity-aware. As written, it routes by `worker.capability` (local vs remote), which is a one-bit decision — but if there are multiple registered remote workers, the seam doesn't say which one wins. **Suggested fix:** phase 2 step 2.5 should pin a degenerate single-worker policy explicitly ("if multiple remote workers are registered, the first one returned by `listWorkers()` wins; phase 3 fixes this") and the test in step 2.7 should *only* register one remote worker. Alternatively, push the registry-aware part of `MultiHarnessPool` into phase 3 and have phase 2 do hardcoded single-remote routing — phase 2 explicitly scopes itself to "single remote worker" already.

- **Phase 3 step 3.2 — extending `RuntimePort.dispatchRun` with `targetWorkerId` breaks the README non-negotiable about transport-agnostic ports.** README (`README.md:16`) says "`RuntimePort`, the IPC frame contracts, and the `Store` port should not grow distributed-system concepts; those live behind concrete transport / registry / lease implementations." Phase 3 step 3.2 (`phase-3:75`) adds `options?: { targetWorkerId?: string; policyHint?: 'sticky' | 'capacity' }` to `dispatchRun` and a `not_dispatchable` variant to `DispatchRunResult`. Both are distributed-system concepts (worker identity, policy hints, capacity-aware failures) bleeding into the port. **Suggested fix:** keep `RuntimePort.dispatchRun` shape stable; the picker should be a *layer above* `RuntimePort`, owned by the scheduler, that calls into a separate `WorkerRegistryPort.selectWorker(scope, hints)` and then `RuntimePort.dispatchRun(...)` with the chosen worker bound through composition (constructor capture), not through a method argument. This keeps the non-negotiable.

- **Phase 4 step 4.7 — flag flips silently retroactively change phase 1's "not yet dispatchable" promise.** Phase 1's exit criteria (`phase-1:269`) state "a registered remote worker... **never** receives a `dispatchRun` call" with the implicit promise that this property survives until phase 2 explicitly removes it. Phase 4 step 4.7 (`phase-4:217-228`) flips six default flags to `true`, including text-phase routing, in one commit with the title "default all feature-phase agents to remote." A user upgrading through commits will see remote dispatch *suddenly* turned on by default at this commit. The phase doc gives no upgrade path for an existing deployment that has remote workers registered for tasks but doesn't want feature-phase agents dispatched there. **Suggested fix:** decouple "the wire path works" (which lands earlier) from "the default is on" (a separate gated flip). Add an upgrade note to `02-distributed/README.md` and to phase 4's exit criteria explicitly stating that pre-existing deployments need to either register feature-phase-capable workers before upgrading, or set `remoteFeaturePhases.*: false` in config first.

- **Phase 4 step 4.8 — `VerificationService.verifyFeature` is deleted but `IntegrationCoordinator` still references it indirectly.** Phase 4 step 4.8 (`phase-4:252`) deletes `verifyFeature`'s body, claiming "the merge queue's `IntegrationCoordinator` does not call `VerificationService` directly". Then the phase says to "confirm via grep before deletion." This is fragile: the merge train executor design memo (`merge_train_executor_design`) says `IntegrationCoordinator` consumes verification *output*, but the coordinator may invoke verify on rebase (post-rebase verification is often required for merge-queue correctness). **Suggested fix:** before step 4.8 lands, add an explicit pre-step that traces every call site of `VerificationService.verifyFeature` and either retargets each to consume a remote-produced `VerificationSummary` or routes the request through the feature-phase dispatch plane. Don't delete based on grep; delete based on a tracing audit. The current step description as written invites a regression where a post-rebase verify suddenly fails because no agent runs on the orchestrator.

- **Phase 5 step 5.3 — heartbeat-driven renewal cannot scale past one connection per worker.** Phase 5 step 5.3 (`phase-5:267-292`) extends `health_pong` (a phase-1 frame, single connection per worker) with `leases: [{ agentRunId, fence }]`. But phase 3 step 3.4 (`phase-3:147`) makes a worker host *multiple concurrent runs*, each in its own scratch directory. The worker-side runtime (which manages those runs) is not necessarily the same process as the registry client (which sends the heartbeat); phase 1 doesn't say. If they're different processes, the registry client has to query each runtime's held runs every heartbeat period, and there's a race where a run completes between query and pong. **Suggested fix:** specify in phase 5 step 5.3 that the worker *registry client* and the *run executor* are the same process, or that the run executor maintains a shared `Set<{agentRunId, fence}>` the registry client reads atomically. Otherwise the design has a TOCTOU bug at heartbeat time.

- **Phase 5 step 5.4 — sweep-on-boot ordering is asserted but not enforced architecturally.** Phase 5 step 5.7 review prompt (`phase-5:498-501`) requires "the double-crash scenario runs the sweep *before* the scheduler tick on boot, so a reattaching worker cannot win a race against takeover." But step 5.4 (`phase-5:332`) only says "start the sweeper interval inside `scheduler.run()`." Nothing pins the *startup* sweep before the first scheduler tick. If `scheduler.run()` immediately ticks before the sweep interval fires, the race exists. **Suggested fix:** step 5.4 should specify that the lease-sweeper's first invocation is synchronous in `scheduler.run()`, before the first tick: `await sweeper.sweep(now); scheduler.tick(); setInterval(() => sweeper.sweep(...), sweepIntervalMs);`. State this explicitly in the step body, not just in the test review.

- **Phase 5 — fence on git push is unforgeable but not on every state-mutating IPC frame, leaving a gap.** Phase 5 step 5.5 (`phase-5:373-379`) classifies `progress` and `assistant_output` as "advisory" and skips fence-checks. But `progress` frames update `agent_runs.run_status` and persist token-usage rollups (per memory `budget_usage_rollup_architecture`); a stale worker's `progress` after takeover would corrupt the run's monotonic state. Similarly, `confirm` (the task progress acknowledgement from worker-model.md) is not in the list. **Suggested fix:** every frame that touches `agent_runs` or downstream rollup tables must carry fence. The advisory exception is too broad. Re-classify per-frame: `progress` (mutates run row) → fenced; `assistant_output` (TUI display only, transient) → unfenced; `confirm` → fenced.

- **Phase 5 step 5.9 — column-drop migration ordering vs phase 3's owner column.** Step 5.9 (`phase-5:578-580`) drops `worker_pid` and `worker_boot_epoch` via the `CREATE TABLE … AS SELECT` rebuild pattern. But the rebuild requires explicitly listing every kept column. If phase 3 step 3.1's `owner_worker_id` is still in the schema (per the duplicate-source-of-truth issue above), the rebuild migration as written will either preserve a now-orphan column or silently drop it. **Suggested fix:** explicitly enumerate what the post-step-5.9 `agent_runs` schema looks like (columns kept, columns dropped, columns introduced by phase 5) so reviewers can match it against phase 3's adds. This is the kind of migration list that should be in the cross-phase conventions section of `README.md`.

- **Phase 4 — bootstrap planner has identity-allocation hand-wave.** Phase 4 step 4.6 (`phase-4:188-213`) says "the bootstrap planner... runs through the same `dispatchRun({ scope: 'feature_phase', phase: 'plan' })` path; identity is allocated by the orchestrator before dispatch... the scheduler creates the placeholder feature when the user kicks off a project." But `RunScope.feature_phase` requires a `featureId` (`phase-4:31` cites `src/runtime/contracts.ts:33-35`), and the orchestrator creates a placeholder feature. What happens if the planner's proposal *renames* the feature, or if the user aborts the bootstrap mid-stream and the placeholder feature has no real plan? `phase-4:189` says it differs only in "no prior proposal events" — not enough to characterize the placeholder's lifecycle. **Suggested fix:** spell out the bootstrap-feature lifecycle in step 4.6, including: how the placeholder feature is named, what happens on bootstrap abort, whether the placeholder is hidden from the TUI feature list, and how rename interacts with `agent_runs.feature_id` FKs.

- **Phase 2 step 2.1 — push restriction by `agent_run_id` push option is forgeable.** Phase 2 step 2.1 (`phase-2:128-131`) says the bare-repo pre-receive hook reads `agent_run_id` from a push option (`-o agent_run_id=<id>`) and the orchestrator answers whether that id maps to that branch. But push options are sent by the worker and are entirely client-controlled. A compromised or misbehaving worker can simply send the right `agent_run_id` for a branch it has no business pushing to — the hook has no signed proof. **Suggested fix:** the pre-receive hook should look up the *current* lease for the branch's task (via phase 5's `run_leases`) and reject pushes whose `agent_run_id` doesn't match the lease holder *and* whose fence (also a push option) isn't current. This depends on phase 5 landing — phase 2 should explicitly mark this as "weak authorization until phase 5; document residual risk in `docs/concerns/`."

## Minor / nits

- **README.md `01-baseline` cross-reference is stale.** README (`README.md:5`) says "this track assumes [01-baseline] has merged." Phase 3 step 3.1 (`phase-3:39`) says "if `01-baseline` has not yet shipped its 010, this phase claims `010`." These two statements together imply 01-baseline can ship after 02-distributed, which contradicts the README assumption. **Suggested fix:** delete the conditional and pin migration numbers as suggested above.

- **Phase 1 step 1.5 — TUI panel path is hand-waved.** `phase-1:222` says "verify path during implementation." Hand-waving the file path in a phase doc invites scope creep at implementation time. **Suggested fix:** spend ten minutes finding the path now.

- **Phase 2 step 2.3 — correlation-id collisions are "impossible" but not bounded.** `phase-2:294` says correlation ids are "random + per-run scope." With high-frequency session ops, even 32-bit collisions are not impossible. **Suggested fix:** make them monotonic per-run instead of random; collision-free by construction.

- **Phase 2 step 2.4 — `RemoteSshHarness` name is misleading.** The transport is WebSocket (chosen in phase 1 step 1.4, `phase-1:77`), not ssh. The bare-repo transport is ssh, but the harness is the IPC plane. **Suggested fix:** rename to `RemoteWsHarness` or just `RemoteHarness` to avoid future confusion.

- **Phase 3 step 3.6 — index migration is **011** but phase 3 step 3.1 already claimed 010, and phase 1 step 1.3 took **012**.** This means phase 3 step 3.1 = 010, phase 3 step 3.6 = 011, phase 1 step 1.3 = 012 — but only if the phases ship in the order phase-3-3.1 → phase-3-3.6 → phase-1, which is impossible since phase 3 depends on phase 1. **Suggested fix:** see the critical issue on migration numbering.

- **Phase 4 step 4.2 — alternative IPC frame design ("add a second `run_feature_phase` variant") is left as a reviewer choice.** `phase-4:84-86` invites the IPC schema reviewer to pick either a discriminated `run` extension or a separate variant. Phase docs should not punt design decisions to later review. **Suggested fix:** make the call now and commit to it.

- **Phase 5 step 5.1 — migration filename is `0NN_run_leases.ts`, literally.** `phase-5:165` says "Pick the next free id." Doc should pick now. **Suggested fix:** state the actual number after coordinating per the critical issue.

- **Phase 5 step 5.5 — push-option fence enforcement requires signed push options.** `phase-5:387-389` says workers pass fence as `--push-option fence=<n>`. Push options are unauthenticated by default; the hook trusts what the client sends. Same forgery concern as the `agent_run_id` issue above. **Suggested fix:** state the trust model explicitly. If we trust the worker (because it's behind ssh-auth from the bare repo), document that. If not, sign push options.

- **Phase 4 step 4.4 — `VerificationLayerConfig` ships in payload but the worker has no way to re-fetch on retry.** `phase-4:131,150` says config ships once per dispatch. If the orchestrator changes config mid-feature (e.g. operator edits verification command list), in-flight runs use stale config. Probably acceptable, but should be stated.

- **Phase 1 step 1.5 — `WorkerRegistryClient` is "the worker-side reference impl" but lives in `src/runtime/registry/`.** `phase-1:220-223` says it's reused by the worker daemon in phase 2. But the orchestrator code shouldn't import worker code; the registry client conceptually belongs in `src/runtime/worker/` or a shared place. **Suggested fix:** put it in `src/runtime/registry/client.ts` and add a comment that orchestrator imports are forbidden (or move it to a shared `src/runtime/registry/` subdirectory after phase 2).

- **Phase 5 step 5.2 review prompt — "grant happens *after* the worker has acked the run" is hard to pin down.** `phase-5:255-258` adds a complicated ordering constraint (grant after worker ack, before runStatus = 'running'). What is "worker ack" precisely — receipt of the `run` frame, or first heartbeat carrying the new lease? **Suggested fix:** define "worker ack" in step 5.2 explicitly. Probably "first `worker_init_ack` from `phase-2:347`" — but that's named in phase 2, not phase 1, so it's a forward dep.

## Cross-phase inconsistencies

- **Phase 1 vs Phase 5 — `bootEpoch` semantics.** Phase 1 step 1.1 (`phase-1:39-44`) uses `bootEpoch` as a "cache-invalidation token" and not part of the registry primary key. Phase 5 doesn't mention `bootEpoch` at all — fence tokens replace it entirely. But phase 5 doesn't say `bootEpoch` is retired or what it means after phase 5 lands. Does a worker restarting (new bootEpoch, same workerId) reclaim its old leases? Phase 5 implies no (heartbeats stop, leases expire), but the column lingers. **Suggested resolution:** phase 5 step 5.9 should include "drop the `boot_epoch` column from `workers`" or explicitly state why it stays.

- **Phase 2 vs Phase 4 — bare-repo branch namespace.** Phase 2 step 2.1 (`phase-2:115-120`) defines two branch-name patterns: feature (`feat-<slug>-<feature-id>`) and task (`feat-<slug>-<feature-id>-<task-id>`). Phase 4 introduces remote feature-phase agents that *write* on a feature branch (the verifier executes `npm test` against the feature worktree, the planner mutates a feature graph proposal). The hook's namespace check (`phase-2:128-131`) only authorizes task-branch pushes for tasks; what authorizes a feature-phase agent's push (or doesn't it push)? Phase 4 says workers "execute commands against its locally-cloned worktree" but doesn't say if results are pushed. Verification is read-only against the worktree, but planner runs may apply schema migrations or generate code. **Suggested resolution:** phase 4 should explicitly state which feature-phase scopes, if any, push commits to the bare repo, and extend the pre-receive hook authorization in step 4.4 if needed.

- **Phase 1 vs Phase 4 — `harnessKinds` capability vs `scopeKinds` capability.** Phase 1 step 1.1 (`phase-1:111`) declares `WorkerCapabilities = { scopeKinds, harnessKinds }`. Phase 2 step 2.5 routes by `harnessKinds` (local-spawn vs remote-ssh). Phase 4 implicitly routes by `scopeKinds` (does this worker do `feature_phase`?). These are two orthogonal capability dimensions, but neither phase 2 nor phase 3 explicitly routes by `scopeKinds`, so a phase 1-registered worker that declared `scopeKinds: ['task']` will silently get feature-phase work in phase 4. **Suggested resolution:** phase 3's picker (step 3.3) should filter by both `scopeKinds.includes(scope.kind)` and `harnessKinds.includes(harnessKind)`. State this explicitly.

- **Phase 3 vs Phase 5 — sticky resume vs lease takeover.** Phase 3 step 3.5 (`phase-3:175-198`) says sticky resume's behavior on `unknown_worker` is "fall through to `mode: 'start'`" — i.e. lose the session. Phase 5 step 5.4 (`phase-5:319-345`) says lease takeover routes via `mode: 'resume'` *with the same `sessionId`* — i.e. the session is preserved across worker change. Once phase 5 lands, the phase 3 sticky-resume "lose-the-session" behavior should be retired (since the session lives on the orchestrator and the new worker can resume it). But phase 5 never says "delete the phase 3 fallback." **Suggested resolution:** phase 5 step 5.4 should explicitly retire the phase 3 step 3.5 fresh-start fallback, or phase 3 should be rewritten in light of the centralized session storage decision so that sticky resume *always* falls through to "any other worker with capacity, same session id."

- **Phase 4 vs Phase 5 — feature-phase recovery uses pid/proc until phase 5.** Phase 4 known-gap (`phase-4:327`) says "Recovery still uses pid/proc liveness for feature-phase runs after this phase. Heartbeats from phase 1 cover liveness while the agent is running, but the orphan-recovery path on orchestrator restart still grep's `/proc/<pid>/environ`." But phase 4 has *moved* feature-phase agents off the orchestrator (the entire point of phase 4). The orchestrator's `/proc/<pid>/environ` probe was meaningful when the agent was an in-process child; for a remote worker, there is no `pid` on the orchestrator side to probe. After phase 4, the pid column for feature-phase runs is meaningless garbage. **Suggested resolution:** phase 4 step 4.8 should retire pid/proc *for feature-phase runs* (set the column NULL on dispatch), and phase 5 step 5.9 retires it for tasks. Otherwise phase 4 leaves a code path that does nothing useful for an entire commit cycle.

- **Phase 1 vs Phase 2 — heartbeat carries no session/run state, but phase 5 needs it to.** Phase 1 step 1.2 (`phase-1:132-148`) defines a heartbeat frame as bare (`workerId`, `bootEpoch`). Phase 5 step 5.3 (`phase-5:268-272`) extends `health_pong` with `leases: [{ agentRunId, fence }]`. But phase 1 step 1.2 says registry frames are a separate `RegistryFrame` union from `WorkerToOrchestratorMessage`. Where does `health_pong` live? Phase 5 cites "baseline phase-1 step 1.4" for the heartbeat pair, which is a *baseline* track artifact, not 02-distributed's phase 1. So 02-distributed phase 1's heartbeat (`heartbeat`/`heartbeat_ack` per `phase-1:140`) and baseline phase 1's `health_ping`/`health_pong` are *different* heartbeat frames in *different* unions. Phase 5 extends the wrong one. **Suggested resolution:** phase 5 step 5.3 should clarify whether it extends 02-distributed's `heartbeat` (registry-plane, no run scope) or baseline's `health_pong` (per-run, on the IPC plane). They're different protocols. Probably it should extend the registry-plane heartbeat, since that's the per-worker (not per-run) channel.

- **Phase 2 vs Phase 5 — `RemoteSshHarness` `onExit` semantics.** Phase 2 step 2.4 (`phase-2:331-333`) wires `SessionHandle.onExit` to *transport-close*: a dropped network is treated as run termination. Phase 5 says a network partition should *not* terminate the run — phase 5 step 5.8 (`phase-5:511-540`) says the worker keeps running, the orchestrator-side lease expires, and takeover happens. But if `onExit` fires on transport-close (phase 2's design), the run is already marked terminated on the orchestrator side before takeover gets a chance. **Suggested resolution:** phase 5 step 5.4 needs to override phase 2's `onExit` semantics: transport close → mark run "owner_unknown" (or similar new sub-state), wait for lease expiry, then route. Or, phase 2's `onExit` should only fire on a *clean* transport close (worker sent `result`/`error` then closed); a *dirty* close should be a no-op until the lease layer sweeps. Phase 5 must explicitly retract phase 2's `onExit`-on-transport-close design.

- **Phase 4 vs README — auth/mTLS hardening track.** README "Out of scope" section says auth hardening is a separate track. Phase 1 step 1.4 (`phase-1:86`) ships a shared bearer token. Phase 2 step 2.1 (`phase-2:108-110`) ships ssh-key auth for the bare repo. Phase 4 then introduces verification and proposal-host on remote workers — both are higher-stakes (verification controls the merge gate; proposal-host shapes the feature graph) — and inherits whatever phase 1 picked. **Suggested resolution:** add a phase 4 explicit acknowledgement that the threat model has gone up and a follow-up to harden auth before phase 4 defaults to remote (`remoteFeaturePhases.* = true`). Currently phase 4 step 4.7 flips defaults without revisiting auth.

## What the plan gets right

- **Bare-repo as the canonical worktree-sync layer** is the right shape: it co-locates ref state with orchestrator SQLite, avoids GitHub rate limits, and gives the merge train a single authoritative source. README cross-cutting concerns section sells this clearly.

- **Single orchestrator authority** is set as a non-negotiable up front (`README.md:14`); this is exactly the right scope guard. Multi-orchestrator HA would explode the design space; saying "no" loud and early is good architecture.

- **Phase 1's "visible but not dispatchable" property** (`phase-1:90-95,267-269`) is a thoughtful seam: the registry is observable end-to-end before anything actually dispatches across the network, which makes phase 2's dispatch wiring testable against a real registry.

- **Phase 4's per-scope flag rollout** (`phase-4:215-225`, the `remoteFeaturePhases` flag map and the per-scope migration commits in step 4.6) is a good de-risking pattern. Each scope is one revertable commit, and the commit ordering (read-only first, verify next, replanner before planner, bootstrap last) reflects real risk.

- **Phase 4 step 4.9's audit guard** (lint rule + at-rest scan + runtime assertion) is the kind of structural enforcement that stops the non-negotiable from quietly drifting back. The triple-layer (compile, runtime, test) is the right level of paranoia for this property.

- **Phase 5's crash matrix** (`phase-5:101-109`) is concrete: it enumerates every interesting failure and points at the test that proves recovery. Most distributed-runtime designs hand-wave this; doing it as a table makes review tractable.

- **Migration retirement of legacy fields with the rebuild pattern** (`phase-5:574-580` referencing `006_rename_feature_ci_to_ci_check.ts`) is the right SQLite hygiene — `ALTER TABLE DROP COLUMN` is unreliable on older builds, and the pattern is already established in the codebase.
