# Knowledge Files

See [ARCHITECTURE.md](../ARCHITECTURE.md) for the high-level architecture index.

## Codebase Map

A lightweight project structure summary written to `.gvc0/CODEBASE.md`. Generated once per orchestrator run (or on demand via `c` key in TUI), and regenerated automatically only when the structural input bundle changes.

Generated by a `light`-tier model call at session start from a deterministic collector-owned input bundle rather than open-ended repo exploration. Contents injected into `WorkerContext.codebaseMap`. Workers use it to orient without spending tool calls reading the filesystem.

Baseline input bundle:
- a pruned root-relative file tree of tracked/unignored project files
- top-level manifests and workspace/config files when present (`package.json`, `pnpm-workspace.yaml`, `turbo.json`, `tsconfig*.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`, etc.)
- deterministic entry-point candidates inferred from manifest fields and common names like `src/index.*`, `src/main.*`, `src/app.*`, `src/server.*`, and `cli.*`
- lightweight architecture/orientation docs such as `ARCHITECTURE.md` when present

The collector should exclude obvious noise such as `.git/`, `.gvc0/`, `node_modules/`, `dist/`, `build/`, `target/`, `.next/`, `coverage/`, virtualenv/cache directories, and other generated output trees. Normal source-file content edits do not require regeneration by themselves; structural changes such as file additions/removals/renames in relevant roots or manifest/workspace changes should update the structure hash and trigger regeneration.

```
.gvc0/CODEBASE.md contents:
- Key directories and their purpose
- Likely entry points
- Conventions (naming, file structure)
- Truncated source/workspace tree
```

## KNOWLEDGE.md

Append-only file at `.gvc0/KNOWLEDGE.md`. Workers read it at task start (injected into system prompt). Workers append to it when they discover recurring patterns, non-obvious constraints, or rules future tasks should follow.

```typescript
// Tool available to all workers
const appendKnowledgeTool: AgentTool = {
  name: "append_knowledge",
  description: "Record a project rule, pattern, or lesson for future tasks.",
  schema: Type.Object({ entry: Type.String() }), // pi-sdk tool schema shape
  execute: async (_, { entry }) => {
    await fs.appendFile(".gvc0/KNOWLEDGE.md", `\n- ${entry}\n`);
    return { content: [{ type: "text", text: "Recorded." }], details: {} };
  },
};
```

The `Type.Object(...)` / `Type.String(...)` examples in this doc intentionally match the schema style expected by pi-sdk's agent tool definitions; gvc0 should follow that same harness/tool-contract style for first-party tools.

## Decisions Register

Append-only file at `.gvc0/DECISIONS.md`. Records significant architectural decisions made during execution. Prevents parallel workers from making contradictory choices.

```typescript
const recordDecisionTool: AgentTool = {
  name: "record_decision",
  description: "Record an architectural decision that other tasks should respect.",
  schema: Type.Object({
    decision: Type.String(),
    rationale: Type.String(),
  }),
  execute: async (_, { decision, rationale }) => {
    const entry = `\n## ${new Date().toISOString()}\n**Decision:** ${decision}\n**Rationale:** ${rationale}\n`;
    await fs.appendFile(".gvc0/DECISIONS.md", entry);
    return { content: [{ type: "text", text: "Decision recorded." }], details: {} };
  },
};
```

Both files are injected into `WorkerContext` under the `standard` and `quality` token profiles. Omitted under `budget`.

`CODEBASE.md` should stay structural/orienting only. It should not attempt to inline full source contents, runtime scheduler state, git history, or the full contents of `KNOWLEDGE.md` / `DECISIONS.md`; those belong to other context layers.
