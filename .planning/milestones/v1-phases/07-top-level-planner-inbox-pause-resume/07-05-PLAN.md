<objective>
Close the last Phase 7 planner safety gap: make edits against actively planning features
visible and deterministic.

The requirements are stronger than a passive warning. REQ-PLAN-07 says that when a
top-level planner proposes an edit on a feature whose feature-level planner is currently
running, the proposal view must flag the collision, and accepting the proposal must cancel
the running feature-level planner so it can rerun on the new feature shape.

Today there is no collision detector, no top-level proposal metadata for collided runs,
and no approval-path logic to reset and rerun the affected feature planner sessions. This
plan adds exactly that behavior without broadening the blast radius to executing tasks.
</objective>

<interfaces>
Current verified analogs and gaps:
- `listAgentRuns()` already exists on the Store port and can locate active feature-phase runs.
- Feature planner runs live under ids like `run-feature:${featureId}:${phase}` and currently use proposal phases `plan` / `replan`.
- Existing proposal approval flow already appends `proposal_applied`, `proposal_rejected`, and `proposal_rerun_requested` events, but there is no collision metadata in the proposal payload.
- `pendingProposalForSelection(...)` and the proposal controller already surface `await_approval` proposals for the selected feature; this is the correct UI seam to extend.
- `feature_phase_rerun_requested` already knows how to delete the old session and reset a feature-phase run to `ready`.
- There is no current distinction between a proposal that touches an idle feature and one that edits a feature with an active planner run.

Collision definition for this plan:
- A collision exists when a top-level proposal includes any op that targets feature `F` (edit, remove, move, split, merge, or task mutation under `F`) AND `listAgentRuns()` contains a `feature_phase` run for `F` in phase `plan` or `replan` whose `runStatus` is not terminal.
- Executing task runs do NOT count as REQ-PLAN-07 collisions; additive-only apply constraints already cover those live-work protections.

Target approval behavior:
1. top-level proposal reaches `await_approval` with collision metadata attached,
2. if rejected: do nothing to collided feature planners,
3. if approved:
   - cancel/reset each collided feature planner run,
   - delete or clear its session according to fresh-rerun semantics,
   - apply the top-level proposal,
   - enqueue replanning on the affected feature(s) as needed.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| top-level proposal -> active feature planner runs | A proposal can invalidate in-flight planner work; acceptance must cancel/reset only the precise collided planner runs |
| approval surface -> scheduler rerun machinery | UI-level approval must not silently drop planner sessions or apply without resetting invalidated runs |
| collision metadata -> operator decision | The operator needs a truthful view of what acceptance will cancel and rerun |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-05-01 | Tampering | Silent planner invalidation | mitigate | Surface collision metadata in approval payload before the operator decides. |
| T-07-05-02 | Denial of Service | Accepted collision leaves stale planner running | mitigate | Approval path explicitly resets collided planner runs and requeues them on the new shape. |
| T-07-05-03 | Repudiation | Unclear which planner runs were cancelled | mitigate | Append explicit collision-resolution events listing affected feature ids/run ids. |
| T-07-05-04 | Elevation of Privilege | Collision handling cancels executing task runs | mitigate | Limit collision scope to active feature planner runs only; executing task runs stay protected by additive-only proposal rules. |
</threat_model>
