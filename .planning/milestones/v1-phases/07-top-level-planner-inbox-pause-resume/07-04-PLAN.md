<objective>
Add the missing planner-session and audit-log layer for the new top-level planner flow.

Phase 07-01 gives the codebase a real top-level planner runtime, but without this plan it
still lacks the two user-facing guarantees required by REQ-PLAN-04 and REQ-PLAN-06:
- re-invoking the planner must let the user choose between continuing the prior chat and
  starting fresh, and
- planner prompts must persist as audit-log context alongside the features they created.

The feature-phase runtime already has the exact low-level analogy: reuse `sessionId` for
ordinary reruns and clear it for fresh reruns. This plan copies that discipline to the
new top-level planner flow and records prompt/session provenance as append-only events.
</objective>

<interfaces>
Current verified analogs and gaps:
- `dispatchFeaturePhaseUnit(...)` already reuses `sessionId` for ordinary reruns and clears it for fresh reruns when a completed proposal phase is rerun.
- `feature_phase_rerun_requested` deletes the old session from `SessionStore` and resets the run row to `ready` with `sessionId: undefined`.
- `PiFeatureAgentRuntime.recordPhaseCompletion(...)` already appends `feature_phase_completed` events with `{ phase, summary, sessionId, extra }`.
- Proposal approval/rejection/rerun already append `proposal_applied`, `proposal_rejected`, and `proposal_rerun_requested` events.
- There is no equivalent top-level planner session registry, no top-level rerun request surface, and no append-only prompt provenance record.
- `handleComposerSubmit(...)` and the current TUI app deps do not expose a top-level continue-vs-fresh chooser yet; Phase 7 only needs the model/API seam, not polished Phase 10 UX.

Target decision shape for this plan:
```typescript
type PlannerSessionMode = 'continue' | 'fresh';

interface TopPlannerDispatchRequest {
  prompt: string;
  sessionMode: PlannerSessionMode;
  runId?: string;
}
```

Target audit model:
- Record the prompt text, session id, session mode, and affected milestone/feature ids as append-only events.
- The prompt record is written when the run is launched, not only after approval.
- Rejection/rerun still keep the prompt history.
- It is acceptable to use `events` as the audit-log store for Phase 7 if the payload shape is explicit and queryable.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| top-level prompt entry -> session persistence | Raw user prompt text becomes durable audit context and must remain correctly attributed to its session/run |
| rerun mode selection -> scheduler dispatch | Continue vs fresh must be applied before the planner run starts so the transcript lineage is truthful |
| audit event reader -> event log | Future readers depend on append-only provenance staying immutable across rejection/rerun cycles |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04-01 | Repudiation | Planner prompt provenance | mitigate | Record prompt/session provenance on launch, not only on accepted proposals. |
| T-07-04-02 | Tampering | Fresh rerun accidentally reuses old transcript | mitigate | Fresh mode explicitly clears/deletes the old session before dispatch. |
| T-07-04-03 | Information Disclosure | Session lineage ambiguity | mitigate | Persist `sessionMode`, `sessionId`, and run id in audit payload so future readers can tell which thread produced which proposal. |
| T-07-04-04 | Elevation of Privilege | UI bypasses scheduler to swap sessions | mitigate | Mode selection feeds orchestrator dispatch only; session mutation stays inside scheduler/store/session-store boundaries. |
</threat_model>
