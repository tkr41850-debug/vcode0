<objective>
Turn the existing append-only `inbox_items` stub into the real Phase 7 inbox model.

Today the code has durable inbox storage, but only as a write-only escape hatch for
semantic failures and destructive approvals. Task help waits (`request_help`) do not
materialize as inbox rows at all, ordinary approvals are only represented on the run row,
and the operator response path is task-id-oriented rather than inbox-oriented. That leaves
REQ-INBOX-01, REQ-INBOX-04, and the model side of REQ-TUI-02 incomplete.

This plan closes that gap in three steps:
1. Widen the Store port so inbox rows can be listed and resolved through the same
   persistence boundary that already owns append.
2. Route help and approval waits into the inbox with structured payloads that match
   what the run row stores.
3. Resolve inbox items by equivalence key so one answer can unblock multiple live waits.

This plan intentionally stops at the model/API surface. Full inbox rendering, filters,
and dedicated inbox navigation remain Phase 8.
</objective>

Current verified gaps:
- `Store` in `src/orchestrator/ports/index.ts` exposes only `appendInboxItem(...)`; there is no read or resolve API.
- `inbox_items` already has `id`, `ts`, `task_id`, `agent_run_id`, `feature_id`, `kind`, `payload`, and nullable `resolution`, plus an unresolved index.
- `request_help` in `src/orchestrator/scheduler/events.ts` only sets `runStatus: 'await_response'` and `payloadJson: JSON.stringify({ query })`; it does not append an inbox row.
- `request_approval` appends an inbox row only for `destructive_action`, leaving `custom` and `replan_proposal` waits invisible to the unified inbox model.
- `compose.ts` exposes `respondToTaskHelp(taskId, ...)` and `decideTaskApproval(taskId, ...)` only; there is no inbox-item-first resolution path and no fan-out semantics.
- `InMemoryStore` in `test/integration/harness/store-memory.ts` mirrors the same append-only limitation, so integration tests cannot exercise list/resolve behavior yet.

Target port shape for this plan:
```typescript
interface InboxItemResolution {
  kind: 'answered' | 'approved' | 'rejected' | 'dismissed';
  resolvedAt: number;
  note?: string;
  fanoutTaskIds?: string[];
}

interface InboxItemRecord {
  id: string;
  ts: number;
  taskId?: string;
  agentRunId?: string;
  featureId?: string;
  kind: string;
  payload: unknown;
  resolution?: InboxItemResolution;
}

interface InboxQuery {
  unresolvedOnly?: boolean;
  kind?: string;
  taskId?: string;
  agentRunId?: string;
  featureId?: string;
}

listInboxItems(query?: InboxQuery): InboxItemRecord[];
resolveInboxItem(id: string, resolution: InboxItemResolution): void;
```

Resolution model for fan-out:
- Equivalence is computed from inbox `kind` plus a canonicalized payload, not from `taskId`.
- `agent_help` rows are equivalent when the normalized query text matches.
- `agent_approval` rows are equivalent when the normalized approval payload matches.
- Existing singleton/system kinds (`semantic_failure`, `merge_train_cap_reached`, etc.) do not fan out unless their payloads also match exactly.
- Resolution records the final fan-out task set in `resolution.fanoutTaskIds`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scheduler wait event -> inbox row | Agent-authored wait payloads cross into durable operator-facing records; payload normalization must prevent fan-out mismatches |
| inbox resolution -> runtime delivery | One operator action can deliver to multiple live workers; only equivalent unresolved waits may receive the same answer |
| compose/TUI -> Store resolution | UI-level helpers must not bypass the Store or leave durable rows inconsistent with delivered responses |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-02-01 | Tampering | Fan-out equivalence matching | mitigate | Match on normalized `kind` + payload only; never match on partial text or on `taskId` alone. |
| T-07-02-02 | Repudiation | Multi-task resolution audit | mitigate | Persist `fanoutTaskIds` and the final resolution kind/note on every affected inbox row. |
| T-07-02-03 | Denial of Service | Invisible task waits | mitigate | `request_help` and all approval kinds must append inbox rows, eliminating hidden manual waits. |
| T-07-02-04 | Elevation of Privilege | UI bypass around unresolved rows | mitigate | All reads/writes go through Store list/resolve helpers; compose delegates instead of mutating raw DB state. |
</threat_model>
