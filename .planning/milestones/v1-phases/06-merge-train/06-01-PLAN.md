<objective>
Make `config.reentryCap` real: wire the already-parsed config field through
`FeatureLifecycleCoordinator` into `MergeTrainCoordinator`, change
`ejectFromQueue` to return a discriminated result so the coordinator can route
cap hits to inbox parking instead of repair tasks, and guard `enqueueFeatureMerge`
so a capped feature cannot re-enter the queue.

Four concrete deliverables:
1. **Cap-aware `MergeTrainCoordinator`.** Constructor accepts `reentryCap?`.
   `ejectFromQueue` returns `'ejected' | 'cap_reached'`. `enqueueFeatureMerge`
   throws `GraphValidationError` when `reentryCount >= cap`.
2. **`failIntegration` parking branch.** Reads `ejectFromQueue` result; on
   `'cap_reached'`, appends `merge_train_cap_reached` inbox item and
   `merge_train_feature_parked` event, returns without creating a repair task.
3. **Config wiring.** `SchedulerLoop` passes `ports.config.reentryCap` to
   `FeatureLifecycleCoordinator`; `FeatureLifecycleCoordinator` forwards it
   to `MergeTrainCoordinator`. All existing call sites that omit the cap arg
   remain valid (`reentryCap` is optional; uncapped behavior is unchanged).
4. **Tests at all three layers.** Core unit tests prove the return value and
   guard semantics. Scheduler-loop unit tests prove the event-driven parking
   path. Integration test proves a parked feature is not re-enqueued and has an
   inbox row in the persisted store.

Purpose: Closes REQ-MERGE-03. Does not touch git, agents, or the integration
runner — those land in 06-02. No new migrations are required because inbox
items already persist via the existing `inbox_items` table (migration 0005)
and `appendInboxItem` on `OrchestratorPorts.store`.

Output: Three modified source files, three extended test files, no new
migrations.
</objective>
