<objective>
Ship the per-task transcript surface (the fourth of Phase 8's four TUI surfaces) together with
the render rate-cap and lightweight render-time virtualization needed to keep the UI responsive
under LLM streaming bursts.

The underlying data already flows end-to-end: worker output arrives via compose.ts →
TuiApp.onWorkerOutput → AgentMonitorOverlay.upsertLog, which maintains a 200-line ring buffer
per agentRunId. The gap is threefold:
1. No task-scoped overlay exists — the existing agent monitor is worker-centric (shows all
   active runs); a transcript surface must be DAG-selection-aware (shows the selected task's run).
2. No rate-cap exists — onWorkerOutput calls this.refresh() on every IPC frame; during
   LLM streaming this can fire many times per second.
3. No keybind or slash command exposes the transcript surface to the operator.

All three are additive to the existing TUI shell. No runtime, store, or compose seam changes.
</objective>
