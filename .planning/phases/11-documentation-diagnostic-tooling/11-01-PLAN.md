<objective>
Ship the first Phase 11 slice by adding a truthful, read-only `gvc0 explain` CLI for feature/task/run state and by reusing the codebase’s existing text-summary logic instead of inventing a second diagnostic model.

The codebase already has the hard parts underneath this slice:
- the Store can read graph snapshots, runs, and events from `.gvc0/state.db`
- `@core/state` already defines the canonical work/collab/run derivations
- the TUI view-model layer already produces human-readable planner-audit, proposal, merge-train, and DAG-node summaries

What is still missing is the operator-facing diagnostic path. Today `src/main.ts` only chooses `interactive` vs `auto`, and `GvcApplication.start()` always shows the TUI. 11-01 should add an explicit pre-TUI `explain` branch that opens project state read-only, prints a concise state-at-a-glance, and exits without starting the orchestrator loop.
</objective>

<interfaces>
Verified starting point:
- `src/main.ts` currently parses only `--auto` and always prints `loading...` before composing and starting the TUI app
- `GvcApplication.start()` always calls `ui.show()` before lifecycle start, so a diagnostic path should branch before app startup rather than extend `AppMode`
- `compose.ts` already owns the project-root bootstrap seam and normalized planner-audit query helper via `listPlannerAuditEntries(...)`
- `@core/state` already owns canonical blocked/presentation/summary derivations
- `src/tui/view-model/index.ts` already contains the most useful human-readable summaries, but many are private helpers nested under overlay-oriented builders
- current persistence supports a current run row plus selected emitted events, not a full event-sourced timeline for every run-state transition

Chosen slice shape:
- add explicit `gvc0 explain feature <id>`, `task <id>`, and `run <id>` dispatch in `src/main.ts`
- bypass `writeStartupNotice()`, `appFactory()`, and `app.start(...)` on the explain path
- add small read-only explain helpers in `src/compose.ts`
- export or lift CLI-usable summary helpers from `src/tui/view-model/index.ts`
- label history as recorded activity and keep it grounded in persisted evidence only
</interfaces>
</context>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI args -> read-only diagnostics | `gvc0 explain ...` must resolve targets and print state without mutating graph/run/event state or starting runtime side effects |
| Store state + events -> human-readable output | current persisted truth becomes operator-facing diagnostic text and must not overclaim unavailable history |
| shared summary helpers -> CLI/TUI parity | extracted text builders will serve two surfaces and must preserve canonical status wording |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01-01 | Tampering | explain dispatch path | mitigate | branch before app startup and keep explain helpers read-only over existing Store APIs |
| T-11-01-02 | Repudiation | recorded activity section | mitigate | label history as recorded activity and limit entries to persisted events plus current run facts |
| T-11-01-03 | Integrity drift | CLI-only formatters | mitigate | reuse `@core/state` derivations and shared view-model text helpers instead of duplicating logic |
| T-11-01-04 | Information disclosure | explain error/output paths | mitigate | print only repo-local state already available in graph/run/event/query seams; avoid raw payload dumps unless already summarized for operators |
</threat_model>
