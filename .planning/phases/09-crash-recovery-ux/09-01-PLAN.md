<objective>
Ship the first concrete Phase 9 slice by replacing the current one-line boot recovery call with a
conservative startup reconciliation pass that understands stale git locks, persisted worker PIDs,
and managed task worktrees.

Today boot only calls `recoverOrphanedRuns()` before the scheduler starts. That preserves the
existing replay/reset logic for task runs, but it does not inspect `agent_runs.worker_pid`, does
not clear stale `.git/index.lock` or `.git/worktrees/*/index.lock`, does not classify managed task
worktrees that have outlived their worker, and does not return any structured recovery report that
later Phase 9 slices can surface in the inbox.

09-01 therefore lands the boot-time substrate, not the full UX:
- conservative lock sweep primitives
- PID reconciliation against persisted `worker_pid`
- managed task-worktree orphan classification
- structured startup recovery reporting
- compose boot ordering that runs all of the above before `scheduler.run()`

This slice keeps the existing replay-backed task-run recovery semantics intact and defers recovery
summary inbox items, explicit orphan actions, `resume_incomplete` operator UX, and kill-9 crash
fault-injection integration coverage to 09-02 / 09-03.
</objective>

- `compose.ts` boot currently does only:
  ```typescript
  start: async () => {
    await recovery.recoverOrphanedRuns();
    await scheduler.run();
    ui.refresh();
  }
  ```
- `RecoveryService.recoverOrphanedRuns()` currently iterates task-scope runs, checkpoints live waits,
  leaves checkpointed waits parked, resumes `running` runs with a `sessionId`, and otherwise resets
  `running` runs back to `ready`
- `WorkerPidRegistry` already exists and exposes `list()` backed by `Store.getLiveWorkerPids()` plus
  OS liveness checks via `isAlive(pid)`
- `GitWorktreeProvisioner.sweepStaleLocks()` currently only clears `.git/worktrees/<name>/locked`
  markers when the referenced `gitdir` target is gone; it does not touch root/worktree `index.lock`
- managed gvc0 worktree paths are canonical under `.gvc0/worktrees/<branch>` via `worktreePath(...)`
- `taskDispatchForRun()` already selects `resume` whenever a run has a `sessionId`; 09-01 should not
  replace that replay path

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted worker PID rows → boot cleanup | recovery uses stored `worker_pid` plus OS liveness checks to decide whether cleanup is safe |
| filesystem lock files → boot mutation | recovery may delete git lock files during startup |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01-01 | Destruction | recovery lock sweep | mitigate | only sweep root/worktree `index.lock` when liveness evidence says the owning managed worker is gone; preserve unrelated/non-gvc0 worktree locks |
| T-09-01-02 | Tampering | stale PID reconciliation | mitigate | classify via `process.kill(pid, 0)` through `WorkerPidRegistry.isAlive()`; on uncertain state, keep the PID/lock rather than deleting it |
| T-09-01-03 | Denial of Service | boot startup path | mitigate | bounded filesystem iteration over managed worktrees only; no polling loops or long retries during boot |
| T-09-01-04 | Integrity drift | task-run recovery | mitigate | keep the existing `recoverOrphanedRuns()` semantics as the inner run-recovery step so 09-01 adds startup orchestration without rewriting replay behavior |
</threat_model>
