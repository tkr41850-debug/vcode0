<objective>
Ship the second Phase 9 slice by making startup respawn truthfully complete: boot should not only reconcile locks and PIDs, it should also re-launch in-flight task work immediately and classify replay outcomes that can never continue.

09-01 already landed the recovery substrate and preserves the old run-recovery behavior. That means boot can attempt replay-backed resume for `running` task runs, but two important gaps remain:

1. Synchronous resume failures (`not_resumable`) are only dropped back to generic `ready` state instead of being fresh-started immediately during recovery.
2. Asynchronous replay failures (`resume_incomplete:*`) currently flow through the generic worker error path, which turns them into `retry_await` churn even though they represent truthful recovery facts like `assistant-text-terminal` or `missing-tool-outputs:*`.

09-02 closes both gaps while keeping scope tight:
- immediate fresh-start fallback for synchronously not-resumable startup resumes
- structured replay diagnostics for already-terminated / missing-tool-output cases
- scheduler handling that parks those replay-incomplete cases with operator-visible inbox diagnostics instead of transient retry loops
- light inbox summary wording so the recovery signal is actually readable in the existing TUI surface

This slice still does not ship the full recovery summary/orphan triage UX from 09-03.
</objective>
