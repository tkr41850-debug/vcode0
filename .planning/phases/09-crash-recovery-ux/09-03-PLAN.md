<objective>
Ship the final Phase 9 slice by closing the operator-facing recovery loop: after boot reconciliation and truthful replay handling have already run, the system should append a single recovery-summary inbox item, expose direct clean / inspect / keep actions for orphan worktrees, and prove the whole restart path with a crash-focused integration test in the normal Vitest lane.

09-01 and 09-02 already land the hard parts underneath this slice:
- boot now reconciles stale locks, dead PIDs, and orphaned managed worktrees
- startup immediately fresh-starts synchronously non-resumable runs
- replay-incomplete resumes park truthfully with structured diagnostics instead of retry churn

What is still missing is the Phase 9 UX contract promised in the roadmap:
1. a single operator-visible recovery summary after startup
2. actionable orphan-worktree inbox items with clean / inspect / keep flows
3. restart-time integration coverage that proves coherent state after a synthetic crash/restart sequence

09-03 should stay narrow and leverage the existing unified inbox surface, TUI command plumbing, worktree cleanup helpers, and file-backed persistence. It should not create a new recovery wizard, new persistence silo, or second replay mechanism.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| startup recovery report -> persisted inbox items | boot-time diagnostics become durable operator-facing rows and must stay concise, truthful, and non-duplicative |
| orphan inbox action -> filesystem mutation | cleanup commands must only touch gvc0-managed task worktrees that startup already classified as dead/absent |
| restart fixtures -> integration assertions | crash-focused tests must exercise real persistence/reopen behavior without depending on external services or PTY infrastructure |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-03-01 | Tampering | orphan cleanup command | mitigate | validate inbox kind/payload before cleanup and limit deletion to the referenced managed task worktree only |
| T-09-03-02 | Repudiation | startup recovery surfacing | mitigate | append explicit recovery/orphan inbox rows and resolve them through the existing inbox resolution path |
| T-09-03-03 | Denial of Service | duplicate startup recovery spam | mitigate | append at most one recovery summary per boot pass and keep per-orphan items bounded to actual orphan findings |
| T-09-03-04 | Information disclosure | inspect action wording | mitigate | show only repo-local branch/path/owner-state facts already present in the recovery report; do not dump unrelated filesystem state |
</threat_model>
