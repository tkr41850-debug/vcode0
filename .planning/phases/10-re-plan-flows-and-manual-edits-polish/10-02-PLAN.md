<objective>
Ship the second Phase 10 polish slice by turning the current sparse approval UX into a real read-only proposal review surface with explicit collision visibility.

The runtime and proposal pipeline already have the hard parts:
- pending top-planner proposals already carry authoritative metadata and collision lists
- `displayedSnapshot(...)` and `ComposerProposalController.getDraftSnapshot()` already provide the live-vs-draft seam
- approval already resets collided feature-planner runs only on accept and emits `proposal_collision_resolved`

What is still missing is the operator-facing review surface. Today the TUI can hint that approval “resets N planner runs”, but it does not clearly show what changes are under review, which planner context produced them, or exactly which live feature-planner runs approval will reset.

10-02 should make proposal review explicit, readable, and collision-complete without adding a second editing model.
</objective>

- add a dedicated read-only proposal review surface for pending planner proposals

- keep approval/reject/rerun flows on the existing command path

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pending proposal payload -> review overlay | the read-only review surface must show the actual proposal under approval, not a shadow interpretation |
| persisted collision metadata -> operator decision | the collision list must reflect the exact `collidedFeatureRuns` stored on the proposal payload |
| review wording -> approval side effects | copy about reset/collision behavior must match the scheduler’s existing accept/reject behavior |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-02-01 | Tampering | proposal preview surface | mitigate | derive preview only from authoritative snapshot, draft snapshot, and persisted proposal payload; no UI-local mutation state |
| T-10-02-02 | Repudiation | collision review | mitigate | render persisted `collidedFeatureRuns` directly and keep integration coverage on accept/reject semantics |
| T-10-02-03 | Information Disclosure | proposal review overlay | mitigate | limit output to repo-local planner prompt/session/scope/collision facts already carried by proposal metadata and graph snapshots |
| T-10-02-04 | Integrity drift | approval side-effect copy | mitigate | keep review text aligned to the existing scheduler path that emits `proposal_collision_resolved` and only resets runs on accept |
</threat_model>
