<objective>
Ship the first Phase 10 polish slice by making top-planner session intent explicit and planner provenance readable.

The underlying behavior is already in the runtime:
- top-planner dispatch and rerun already honor `PlannerSessionMode`
- fresh reruns already clear/delete the previous persisted planner session
- prompt/session provenance already lives in append-only events plus top-planner proposal metadata

What is still missing is the operator-facing TUI layer. Right now a user can trigger top-level planning or rerun a pending top-planner proposal, but the TUI does not clearly surface whether that action continues the prior chat or starts fresh, and it has no readable audit-log surface for the prompts and outcomes that produced current planner state.

10-01 should close exactly those gaps without changing planning semantics, approval semantics, or persistence shape.
</objective>

- `requestTopLevelPlan(prompt, options?)` and `rerunTopPlannerProposal(event?)` already accept `sessionMode?: PlannerSessionMode` through `TuiAppDeps` and `compose.ts`

Chosen slice shape:
- add a dedicated planner-session picker overlay for top-planner submit/rerun intent
- add a normalized planner-audit reader overlay derived from existing events + metadata

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| top-planner submit/rerun action -> session selection | the user-visible continue/fresh choice must map exactly to the real session-store behavior |
| append-only planner events -> audit reader | low-level planner events become human-readable audit rows and must stay truthful to the original provenance |
| selected feature context -> filtered audit view | feature-scoped audit reading must filter existing metadata/events, not infer a second planner history model |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01-01 | Repudiation | planner session picker | mitigate | route continue/fresh strictly through the existing `sessionMode` contract so UI wording and persisted transcript behavior cannot drift |
| T-10-01-02 | Tampering | planner audit reader | mitigate | derive rows from append-only events and persisted metadata only; do not add editable history state |
| T-10-01-03 | Information Disclosure | audit filtering | mitigate | show only prompt/session/scope/outcome facts already present in planner events and metadata, scoped to the current planner context or touched feature ids |
| T-10-01-04 | Integrity drift | transient TUI picker state | mitigate | keep pending choice state ephemeral and dispatch immediately through compose/orchestrator boundaries once selected |
</threat_model>
