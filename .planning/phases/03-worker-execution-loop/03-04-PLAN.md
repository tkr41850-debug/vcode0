```typescript
import type { BeforeToolCallContext, BeforeToolCallResult } from '@mariozechner/pi-agent-core';

export const DESTRUCTIVE_PATTERNS: ReadonlyArray<{ pattern: RegExp; label: string }> = [
  { pattern: /\bgit\s+push\s+.*(--force|-f)\b/, label: 'git push --force' },
  { pattern: /\bgit\s+branch\s+.*(-D|--delete\s+--force)\b/, label: 'git branch -D' },
  { pattern: /\bgit\s+reset\s+.*--hard\b/,                   label: 'git reset --hard' },
];

export function describeDestructive(cmd: string): { label: string } | null {
  for (const { pattern, label } of DESTRUCTIVE_PATTERNS) {
    if (pattern.test(cmd)) return { label };
  }
  return null;
}

export async function destructiveOpGuard(
  ctx: BeforeToolCallContext,
): Promise<BeforeToolCallResult | undefined> {
  if (ctx.toolCall.name !== 'run_command') return undefined;
  const cmd = (ctx.args as { command?: string } | undefined)?.command;
  if (typeof cmd !== 'string') return undefined;
  const match = describeDestructive(cmd);
  if (!match) return undefined;
  return {
    block: true,
    reason: `destructive_op_requires_approval: ${match.label}`,
  };
}
```

Approval round-trip flow (already present, restate for clarity):
1. `beforeToolCall` returns `{ block: true, reason }`.
2. pi-sdk emits a tool-error result with `reason`.
3. Worker separately invokes `ipc.requestApproval({ kind: 'destructive_action', description: cmd, affectedPaths: [...] })`.
4. Parent's approval handler calls `Store.appendInboxItem({ kind: 'destructive_action', payload: { cmd, reason } })`.
5. Parent resolves the approval promise with `granted: boolean`.
6. If granted → worker clears the guard for this tool call only (via a one-shot bypass token) OR simply lets the agent retry the command (pi-sdk will call `beforeToolCall` again — see Task 3 for the one-shot bypass design).

Design: the guard must BOTH block the tool AND trigger the approval round-trip. pi-sdk `beforeToolCall` returning `{ block: true }` stops the tool from executing but does NOT itself invoke `requestApproval`. The cleanest wiring:

```typescript
const agent = new Agent({
  beforeToolCall: async (ctx, signal) => {
    const guardResult = await destructiveOpGuard(ctx);
    if (guardResult?.block) {
      // Fire-and-forget the approval round-trip. The tool call is already
      // blocked; we append an inbox_item via ipc.requestApproval so the
      // user can act out-of-band. When approved later, the agent's retry
      // of the same command will re-invoke beforeToolCall; a one-shot
      // bypass token (see below) lets it pass through.
      const cmd = (ctx.args as { command?: string }).command!;
      const descriptor = describeDestructive(cmd)!;
      queueMicrotask(async () => {
        try {
          const approval = await ipc.requestApproval({
            kind: 'destructive_action',
            description: cmd,
            affectedPaths: [workerCwd], // best-effort: destructive git ops affect the worktree
          });
          if (approval.granted) {
            bypassOnce.set(ctx.toolCall.id, true);
          }
        } catch { /* Phase 7 will materialize a richer UX */ }
      });
    }
    return guardResult;
  },
  // ...existing options
});
```

```markdown
# Destructive ops: non-git commands not yet guarded

**Status:** documented concern, deferred to Phase 7+
**Introduced:** Phase 3 (worker execution loop)
**Owner:** (current Phase 3 planner)

## What's not guarded

Phase 3's `destructiveOpGuard` only blocks three git patterns:

- `git push --force` / `git push -f`
- `git branch -D` / `git branch --delete --force`
- `git reset --hard`

The following destructive shell operations are NOT currently blocked and will
execute if the AI emits them via `run_command`:

- `rm -rf <path>` — even `rm -rf /` (restrained only by `run_command` cwd; if the
  AI cd's out first, it can affect the whole filesystem)
- `find <path> -delete`
- `dd if=... of=...` (overwriting block devices or files)
- `mkfs.*` (formatting filesystems)
- `truncate <path>` (zero-length any file)
- Arbitrary `sudo *` (mitigated only by no-root execution convention)
- `chmod -R 000 <path>` (locking out access)

## Why not fixed in Phase 3

- CONTEXT §D explicitly scoped destructive-op detection to git operations.
- The `rm -rf` pattern is hard to distinguish from legitimate workspace cleanup
  (e.g., `rm -rf node_modules`) without more sophisticated heuristics.
- Sandboxing (worker cwd constraints, UID separation) is the stronger defense
  and Phase 3 partially addresses via `resolveInsideWorkdir` — but `run_command`
  runs `sh -c` with only cwd set to the worktree; `cd /` escapes that.

## Phase 7 expansion

When the inbox UI materializes, extend `destructiveOpGuard` with:

- `rm -rf`, `find -delete`, `dd`, `mkfs`, `truncate`, and `chmod -R 000` patterns.
- Path-aware heuristics: allow `rm -rf <x>` if `<x>` resolves inside the worktree;
  escalate otherwise.
- A user-editable allow-list of "safe" destructive patterns (e.g.,
  `rm -rf node_modules` may be pre-approved by policy).

## Related

- `src/agents/worker/destructive-ops.ts` — the current guard.
- RESEARCH.md §Destructive-Op Detection — full scope discussion.
- RESEARCH.md §Open Questions #1 — the deferral decision.
```
