<objective>
Implement the two-tier wait lifecycle promised by Phase 7: a short hot window keeps the
worker process alive for quick replies, then expiry checkpoints the run, releases the
process, and later respawns through the Phase 3 replay strategy.

The codebase already has the hard primitives — `FileSessionStore`, `resume(...)`, and the
file-backed tool-output store — but they are not wired end-to-end. Today the worker only
saves the transcript at run end, never records tool outputs, and has no notion of a
checkpointed wait distinct from a live `await_response` / `await_approval` run. That means
REQ-INBOX-02 and REQ-INBOX-03 are still conceptual only.

This plan closes that gap by:
1. introducing explicit persisted checkpointed-wait state,
2. saving transcript/tool-output data during the run,
3. releasing live workers after hot-window expiry, and
4. respawning them through the existing resume facade when the operator responds.
</objective>

<interfaces>
Current verified gaps:
- `AgentRunStatus` has only `ready | running | retry_await | await_response | await_approval | completed | failed | cancelled`; there is no persisted status for a checkpointed wait.
- `deriveTaskBlocked(...)` and the TUI view-model only recognize `await_response`, `await_approval`, and `retry_await` as run-blocked states.
- `RecoveryService.shouldResumeTaskRun(...)` resumes only `running`, `await_response`, and `await_approval`.
- `WorkerRuntime` saves the transcript only once, after prompt/continue settles; `handleAgentEvent(...)` emits UI/progress output but does not checkpoint session state.
- `ToolOutputStore.record(...)` exists, but production runtime never calls it.
- `request_help` and `request_approval` tool implementations discard `_toolCallId`, so the system currently cannot relate an unresolved wait to the tool call whose result must be spliced back during replay.
- `LocalWorkerPool` has no concept of releasing a live session while retaining resumable state for later operator resolution.

Required state distinction for this plan:
```typescript
export type AgentRunStatus =
  | 'ready'
  | 'running'
  | 'retry_await'
  | 'await_response'
  | 'await_approval'
  | 'checkpointed_await_response'
  | 'checkpointed_await_approval'
  | 'completed'
  | 'failed'
  | 'cancelled';
```

Equivalent metadata shape if encoded in payload instead of new statuses is acceptable only if all of the following remain true:
- blocked-state derivation can distinguish live vs checkpointed waits without inspecting transport state,
- recovery can resume checkpointed waits deterministically,
- inbox resolution knows whether to deliver to a live session or respawn first.

The preferred path is explicit statuses because current code already keys behavior off `runStatus` in `core/state`, `RecoveryService`, `compose.ts`, and the TUI view model.

Replay contract to preserve:
- unresolved wait tool calls must persist their `toolCallId` and enough payload to later synthesize the tool result;
- when the operator answer arrives after checkpointing, the system must:
  1. load the saved transcript,
  2. load persisted tool outputs,
  3. respawn through `PiSdkHarness.resume(...)`,
  4. deliver the help/approval response,
  5. let the worker continue to a new steady state.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| live worker -> persisted checkpoint | Mid-run transcript and tool-output snapshots become the source of truth for later replay |
| checkpointed wait -> respawned worker | Operator answers delivered after process release must reach the correct respawned run, not a stale or wrong task |
| replay facade -> persisted tool outputs | Missing or mismatched tool outputs could fabricate transcript history if not handled strictly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03-01 | Tampering | Replay transcript reconstruction | mitigate | Persist actual tool outputs keyed by original `toolCallId`; never synthesize missing results. |
| T-07-03-02 | Denial of Service | Lost waits after hot-window expiry | mitigate | Explicit checkpointed run statuses plus recovery/compose support ensure expired waits remain resumable and visible. |
| T-07-03-03 | Repudiation | Operator answer delivered to wrong run after respawn | mitigate | Tie respawn + delivery to persisted run id, session id, and inbox item/task linkage; record resolution after successful delivery. |
| T-07-03-04 | Denial of Service | Transcript saved only at terminal completion | mitigate | Save at `message_end` / `turn_end` so expiry/crash can replay from a recent checkpoint. |
</threat_model>
